## User Story

Como utilizador da aplicação num dispositivo móvel (320px–767px), quero que a interface seja totalmente funcional e legível sem scroll horizontal, para que possa importar CSVs e enviar notas sem precisar de um computador.

Como utilizador em tablet (768px–1023px) ou desktop (1024px+), quero que o layout aproveite o espaço disponível de forma proporcional, para que a experiência visual seja confortável em qualquer ecrã.

---

## Acceptance Criteria

### AC1 — Layout funcional em mobile (320px)
Dado que a aplicação é aberta num ecrã de 320px de largura,
Quando o utilizador navega por todas as secções,
Então não existe scroll horizontal e todos os elementos (formulários, botões, tabela, textarea) são visíveis e utilizáveis dentro do viewport.

### AC2 — Layout funcional em tablet (768px)
Dado que a aplicação é aberta num ecrã de 768px de largura,
Quando o utilizador navega por todas as secções,
Então o layout adapta-se ao espaço disponível sem overflow horizontal e os elementos mantêm proporções adequadas.

### AC3 — Layout funcional em desktop (1024px+)
Dado que a aplicação é aberta num ecrã de 1024px ou mais,
Quando o utilizador navega por todas as secções,
Então o layout usa o espaço disponível com `max-width` adequado e margens laterais proporcionais, mantendo o design original das stories 4.1–4.4.

### AC4 — Media queries usando CSS custom properties dos tokens (Story 4.4)
Dado que os tokens de design estão definidos no `:root` (Story 4.4),
Quando as media queries são implementadas,
Então os valores de padding, margin, gap e dimensões dentro das media queries devem referenciar as CSS custom properties existentes (`var(--space-*)`, `var(--color-*)`) e não valores hardcoded.

### AC5 — Navegação/header acessível em mobile
Dado que o `<header>` com `<h1>` existe na página,
Quando a página é carregada em mobile (320px),
Então o título é legível (sem overflow, sem corte), com padding lateral mínimo de `var(--space-sm)` para não encostar às bordas do ecrã.

### AC6 — Imagens e tabelas responsivas (não extravasam o container)
Dado que existe uma tabela `#matchTable` e um elemento `<img id="qrImage">`,
Quando são visualizados em mobile,
Então a tabela está dentro de `.table-wrap` com `overflow-x: auto` (já existente) e a imagem QR tem `max-width: 100%` para não extravazar o container pai.

- Novos componentes ou redesign de elementos existentes
- Animações ou transições
- Mudanças ao `public/index.html`
- CSS Grid layout (usar apenas Flexbox já presente)
- Breakpoints adicionais (ex: 480px, 1440px)
- Testes automatizados de viewport (fora do âmbito Node.js)

## Dev Notes

### Estado actual de `public/styles.css` (antes desta story)

- **195 linhas** com tokens `:root`, base styles, componentes
- `main` tem `max-width: 980px; margin: var(--space-lg) auto; padding: var(--space-md) var(--space-lg)` — funciona em desktop mas em mobile ocupa 100% sem padding lateral
- `section` tem `padding: var(--space-md); margin-bottom: var(--space-md)` — adequado para mobile, manter
- `button` tem `padding: var(--space-sm) 12px` — em mobile botões lado a lado em `.checks` já usam `flex-direction: column` (ok)
- `.checks` já tem `display: flex; flex-direction: column; gap: var(--space-sm)` — ok em mobile
- `textarea { width: 100% }` — já responsivo
- `<img id="qrImage">` tem `style="max-width: 280px"` em inline style no HTML — em mobile 280px pode extravazar um container de 320px com padding; adicionar `max-width: 100%` via CSS para sobrepor em mobile
- `meta viewport` já definido no `<head>`: `width=device-width, initial-scale=1.0` — ok

### Estrutura de breakpoints a implementar

```css
/* Mobile first: breakpoint max-width 767px */
@media (max-width: 767px) {
  main { margin: var(--space-sm); padding: var(--space-sm); }
  #qrImage { max-width: 100%; }
  /* ajustes adicionais se necessário */
}

/* Tablet: 768px–1023px */
@media (min-width: 768px) and (max-width: 1023px) {
  main { max-width: 100%; margin: var(--space-md) auto; }
}

/* Desktop 1024px+: estilos base já cobrem este breakpoint */
```

### Decisão sobre `#qrImage` inline style
O HTML tem `style="max-width: 280px; display:none;"` inline. A media query em CSS tem maior especificidade quando combinada com o seletor correto (sem `!important` se possível). Alternativa: usar `#qrImage { max-width: 100% }` na media query (especificidade de ID sobrepõe inline style para `max-width`? Não — inline tem maior prioridade). Solução: usar `max-width: 100% !important` na media query mobile OU aceitar 280px em mobile 320px com `.table-wrap`-like wrapping. Avaliação no momento da implementação.

### Decisions Log

1. **`#qrImage` inline style override:** HTML tem `style="max-width: 280px"` inline. Inline styles têm maior especificidade que qualquer seletor CSS. Decisão: usar `max-width: 100% !important` na media query mobile para garantir que a imagem não extravasa em 320px. Justificação: `!important` é aceite aqui pois é um override explícito de um inline style de terceiro que não pode ser editado sem alterar o HTML (fora de scope).
2. **`main { border-radius: 0 }` em mobile:** Em 320px, border-radius no `main` cria espaço visual desperdiçado nas bordas. Decisão: remover border-radius em mobile para maximizar área útil.
3. **`button { width: 100% }` em mobile:** Botões com largura fixa são difíceis de tocar em mobile. Decisão: expandir para 100% com `box-sizing: border-box` para manter o padding. Aplicado a todos os botões em mobile — coerente com UX mobile-first.
4. **`input[type="file"] { width: 100% }` em mobile:** Sem este override, o input de ficheiro pode transbordar o container em 320px. Adicionado por precaução.
5. **Desktop (1024px+):** Estilos base no CSS já cobrem este breakpoint adequadamente (`max-width: 980px; margin: var(--space-lg) auto`). Não foi necessário adicionar bloco `@media (min-width: 1024px)` explícito — AC3 cumprido pelos estilos existentes.
