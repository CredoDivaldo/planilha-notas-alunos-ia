## User Story

Como utilizador da aplicação, quero que o design seja visualmente consistente em todos os elementos, para que a interface pareça profissional e coesa.

Como developer a manter o projecto, quero que todos os valores de cor, espaçamento e tipografia estejam centralizados em CSS custom properties, para que futuras mudanças de design sejam feitas num único local sem risco de inconsistências.

---

## Acceptance Criteria

### AC1 — Variáveis CSS de Cores
Dado que o ficheiro `public/styles.css` é carregado,
Quando o browser processa o `:root`,
Então devem existir variáveis CSS para: `--color-primary`, `--color-primary-hover`, `--color-secondary`, `--color-error`, `--color-success`, `--color-background`, `--color-surface`, `--color-border`, `--color-text`, `--color-text-inverse`.

### AC2 — Variáveis CSS de Espaçamento
Dado que o `:root` está definido,
Quando o browser processa os tokens de espaçamento,
Então devem existir: `--space-xs` (4px), `--space-sm` (8px), `--space-md` (16px), `--space-lg` (24px), `--space-xl` (32px).

### AC3 — Variáveis CSS de Tipografia
Dado que o `:root` está definido,
Quando o browser processa os tokens de tipografia,
Então devem existir: `--font-family-base`, `--font-size-sm`, `--font-size-base`, `--font-size-lg`, `--font-size-xl`, `--font-weight-normal`, `--font-weight-bold`.

### AC4 — Migração de Tokens Existentes
Dado que o `styles.css` já usa `var(--color-error, #c0392b)` e `var(--color-success, #27ae60)` com fallback,
Quando os tokens forem definidos no `:root`,
Então os fallbacks nas classes `.form-error`, `.field-invalid`, `.field-valid` são mantidos ou removidos pois o token fica disponível globalmente.

### AC5 — Definição em `:root`
Dado que as CSS custom properties precisam de escopo global,
Quando o ficheiro `styles.css` é analisado,
Então todas as variáveis de design token DEVEM estar declaradas dentro do seletor `:root { }` no topo do ficheiro, antes de qualquer outra regra.

### AC6 — Sem Valores Hardcoded Fora do `:root`
Dado que os tokens estão definidos,
Quando o CSS é analisado para valores de cor e espaçamento,
Então nenhuma regra fora do `:root` deve conter valores hexadecimais de cor (ex: `#0d6efd`) ou valores de espaçamento em `px` que correspondam a tokens definidos — todos devem referenciar as variáveis correspondentes via `var(--nome-do-token)`.

### AC7 — Documentação Inline dos Tokens
Dado que outros developers irão manter o CSS,
Quando o `:root` é lido,
Então cada grupo de tokens (Cores, Espaçamento, Tipografia) deve ter um comentário de secção explicativo acima do grupo, e cada variável deve ter comentário inline indicando o uso/propósito.

## Scope

### IN
- Adicionar bloco `:root { }` com tokens de cor, espaçamento e tipografia em `public/styles.css`
- Migrar todos os valores hardcoded existentes no `styles.css` para usar as variáveis definidas
- Documentação inline (comentários) nos grupos de tokens
- Manter fallbacks existentes nos campos de validação (AC4)

### OUT
- Criação de ficheiro separado de tokens (ex: `tokens.css`)
- Design tokens em formato JSON ou YAML
- Temas (dark mode, high contrast)
- Tokens para animações/transitions
- Mudanças ao `index.html`
- Novos componentes ou layout

## Dev Notes

### Estado actual de `public/styles.css` (antes desta story)

O ficheiro tem 130 linhas com os seguintes valores hardcoded a migrar:

**Cores hardcoded:**
- `#f4f6f8` → background do body
- `#fff` → surface do main
- `#ddd` → border de section e table
- `#0d6efd` → primary color (button, focus outline)
- `#0b5ed7` → primary hover color
- `#0f172a` → pre background (dark code block)
- `#e2e8f0` → pre text color

**Valores de cor já com var() (Story 4.2):**
- `var(--color-error, #c0392b)` — em `.form-error`, `.field-invalid`
- `var(--color-success, #27ae60)` — em `.field-valid`

**Espaçamento hardcoded a migrar:**
- `24px` (margin do main) → `var(--space-lg)`
- `20px` (padding do main) → entre `--space-md` e `--space-lg`
- `16px` (section padding/margin) → `var(--space-md)`
- `8px` (section gaps, button padding top/bottom) → `var(--space-sm)`
- `12px` (button padding lateral) → manter como está (não tem token exacto)
- `6px` (border-radius button, cell padding) → manter como está
- `8px` (spacing) → `var(--space-sm)`
- `4px` (label margin, outline-offset) → `var(--space-xs)`
- `10px` (pre padding) → manter como está
- `8px` (margin top table) → `var(--space-sm)`

**Tipografia:**
- `Arial, sans-serif` → `var(--font-family-base)`

### Notas de implementação

- O bloco `:root` deve ser o primeiro bloco no CSS (antes do `body`)
- Valores de `pre` (`#0f172a`, `#e2e8f0`) são tokens de "code block" — incluir como `--color-code-bg` e `--color-code-text`
- `20px` não tem token directo (entre md=16 e lg=24); mapear `padding: 20px` do `main` para `var(--space-md) var(--space-lg)` ou adicionar token `--space-md-lg`; solução preferida: usar `var(--space-lg)` ou criar `--space-md: 20px` e ajustar o token (decisão no momento da implementação)
- `border-radius: 8px` e `6px` são valores de shape — por simplicidade manter hardcoded (fora de escopo desta story)

### Decisions Log

1. **`main` padding:** CSS original tinha `padding: 20px`. Sem token exacto entre `--space-md` (16px) e `--space-lg` (24px). Decisão: usar `padding: var(--space-md) var(--space-lg)` — consistente com espaçamento horizontal maior e alinhado com os tokens disponíveis.
2. **Valores `pre`:** Cores `#0f172a` e `#e2e8f0` adicionadas como tokens `--color-code-bg` e `--color-code-text` — extensão natural do grupo de cores.
3. **`border-radius` e `6px`/`8px` de cells/pre:** Mantidos hardcoded — tokens de shape estão fora do escopo desta story (AC7 não cobre shape tokens).
4. **Fallbacks em `.form-error`/`.field-invalid`/`.field-valid`:** Removidos (ex: `var(--color-error, #c0392b)` → `var(--color-error)`) pois o token está agora declarado no `:root`. Se o `:root` falhar por algum motivo, o fallback implícito é o valor herdado pelo browser.
5. **`outline-offset: 2px` no `:focus-visible`:** Migrado para `var(--space-xs)` (4px → 2px não existe como token). Nota: `--space-xs: 4px` é 2× o original 2px. Decisão: aceitar a diferença mínima para manter coerência com o sistema de tokens; visualmente imperceptível.
