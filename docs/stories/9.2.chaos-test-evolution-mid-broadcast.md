## Story Statement

**As a** operador,
**I want** testar o comportamento do sistema quando a Evolution API é interrompida no meio de um broadcast,
**so that** eu valide recuperação, sanitização de erros e preservação da lógica de broadcast/status.

---

## Acceptance Criteria

- [ ] **AC1:** Iniciar um broadcast com 5+ destinatários usando o fluxo existente.
- [ ] **AC2:** Matar/interromper o container/processo da Evolution API mid-broadcast (ex.: `docker kill evolution_api`).
- [ ] **AC3:** FastAPI devolve erro HTTP **502** com erro sanitizado (sem leak de internals), alinhado ao comportamento definido em Story 3.6.
- [ ] **AC4:** O status do broadcast fica recuperável: a lógica de retry permite concluir (ou reprocessar) sem ficar em estado irrecuperável.
- [ ] **AC5:** Validar edge case de rate limiting: 11ª mensagem no mesmo dia (ou limite configurado) produz comportamento esperado (HTTP 429 ou resposta educada, sem quebrar o broadcast inteiro).

---

## Scope

### IN

- Chaos injection mid-broadcast
- Verificação de resposta HTTP e recuperação/retry
- Verificação do edge case de rate limiting

### OUT

- Rebuild de broadcast engine
- Migração/upgrade da Evolution API
