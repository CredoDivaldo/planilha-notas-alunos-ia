## Story
**As a** professor,
**I want** o estado da ligação WhatsApp a actualizar-se automaticamente,
**so that** não precise de refrescar a página manualmente para ver se o QR Code foi escaneado.

## Acceptance Criteria

- [x] **AC-1 — Polling inicia ao exibir QR:** Quando o QR Code é exibido no `<img id="qrImage">`, o polling arranca automaticamente com intervalo de 5 segundos, sem intervenção manual.
- [x] **AC-2 — Polling para ao atingir estado "open":** Quando `/api/evolution/instance/state` retorna `instance.state === "open"`, o `clearInterval` é chamado, o QR é ocultado e o `#evolutionStatus` mostra "Ligado ao WhatsApp."
- [x] **AC-3 — Polling para em caso de erro:** Uma falha de rede ou resposta HTTP de erro interrompe o ciclo (clearInterval) e apresenta mensagem de erro no `#evolutionStatus`, sem entrar em loop infinito.
- [x] **AC-4 — Indicador visual de estado durante polling:** O `#evolutionStatus` alterna entre "A verificar ligação... (tentativa N/24)", "Ligado ao WhatsApp." e "Erro: <mensagem>" conforme o ciclo avança.
- [x] **AC-5 — QR actualiza se base64 mudar:** Se `/api/evolution/instance/connect` retornar um novo `qrcode.base64` (diferente do anterior), o `#qrImage.src` é actualizado em tempo real.
- [x] **AC-6 — Limpeza garantida (clearInterval):** O intervalId é guardado em variável de módulo. Antes de iniciar novo polling, qualquer intervalo anterior é cancelado. Ao desligar/fechar a página (`window.beforeunload`), o intervalo é também limpo.
- [x] **AC-7 — Timeout máximo de 2 minutos:** Após 24 iterações (24 × 5 s = 120 s), o polling para automaticamente com a mensagem "Tempo esgotado. Sem ligação após 2 minutos. Tenta novamente."
- [x] **AC-8 — Estado inicial antes do polling:** Um fetch único a `/api/evolution/instance/state` é efectuado antes de iniciar o intervalo, de modo a mostrar o estado actual imediatamente ao professor.

### O que Falta Implementar

1. **`setInterval`-based polling** com `pollingIntervalId` de módulo — núcleo desta story
2. **`stopPolling(reason)`** com 3 motivos: `"connected"`, `"error"`, `"timeout"`, `"unload"`
3. **Timeout de 2 minutos** (24 iterações × 5 s) — a `waitForQr` actual só espera 30 s
4. **Limpeza no `beforeunload`** — a implementação actual não limpa o timer
5. **Actualização do QR em tempo real** — a `waitForQr` actual sai ao obter o primeiro QR; esta story continua a verificar se o QR muda
6. **Fetch de estado inicial** antes de arrancar o intervalo

### Padrão de Fetch existente (reutilizar)

```js
async function callEvolution(endpoint, method = "GET") {
  const response = await fetch(endpoint, { method, headers: { "X-API-Key": window.API_KEY || "" } });
  const payload = await response.json();
  if (!response.ok) throw new Error(payload.error || "Falha na Evolution API");
  evolutionResult.textContent = JSON.stringify(payload, null, 2);
  return payload;
}
```

O `startPolling` deve reutilizar `callEvolution` para manter consistência de autenticação e logging.

### Nota sobre `aria-live`

`#evolutionStatus` em `index.html` linha 64 usa `role="alert"` + `aria-live="assertive"`. Para mensagens de polling repetitivas (a cada 5 s), avaliar downgrade para `aria-live="polite"` — apenas após confirmar com Story 4.1 que não quebra outros fluxos de alerta.

### Issues Found

| # | Severity | Category | Description |

| QA-4.3-001 | LOW | Code | `startPolling` internally calls `stopPolling("unload")` to cancel a previous interval. Using the `"unload"` reason here is semantically misleading — the intent is "cancel-before-restart", not "page is closing". The skip on DOM updates is incidentally correct but relies on an implicit side-effect of the `"unload"` branch. |
| QA-4.3-002 | LOW | Tests | `tests/public/polling.test.js` re-implements `startPolling` / `stopPolling` inline instead of importing from `public/app.js`. If production logic diverges, tests continue passing while the real code breaks. Consider extracting the module as a pure function file or using `jest.mock` / JSDOM script injection. |
| QA-4.3-003 | LOW | Requirements | AC-7 specifies 24 iterations × 5 s = 120 s. The guard `iteration > maxIterations` (strict greater-than) fires on iteration 25, yielding 125 s. Change to `iteration >= maxIterations` to align with the 120 s spec, or update the AC to document the actual 125 s behaviour. |
