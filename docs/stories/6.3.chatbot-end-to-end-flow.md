## Story

**As a** student,  
**I want** to receive a WhatsApp reply to my question about my grades within seconds,  
**so that** the full chatbot flow works reliably end-to-end.

---

## Acceptance Criteria

### AC-1: Full flow delivers AI reply via WhatsApp
**Given:** a student sends a WhatsApp message to the platform number  
**When:** the webhook receives the message, identifies the student and generates an AI response  
**Then:** the system sends the AI-generated reply to the student via Evolution API and the full flow completes without error

### AC-2: Rate limiting blocks excessive messages
**Given:** a student has already sent 10 messages today (configurable via `CHATBOT_RATE_LIMIT_DAILY`)  
**When:** the student sends an 11th message  
**Then:** the system does NOT call the AI API, sends a polite rate-limit message ("Atingiste o limite de mensagens para hoje. Tenta amanhã.") and logs the rate limit event

### AC-3: AI failure does not crash the system or leave the student without reply
**Given:** the AI API returns an error  
**When:** the chatbot attempts to respond  
**Then:** the system sends the fallback message to the student via WhatsApp and records the error in logs

### AC-4: Unknown student receives a clear message
**Given:** the sender phone number is not registered  
**When:** the webhook processes the message  
**Then:** the system sends a reply: "O teu número não está registado. Fala com o teu professor." and does not expose any data

### AC-5: Each chatbot interaction is logged
**Given:** any message is processed by the chatbot  
**When:** the interaction completes (success, fallback, rate limit or unknown number)  
**Then:** a log entry is recorded with: timestamp, normalised phone, outcome type, student ID (if identified) and whether AI was called

### AC-6: Node legacy flows are unaffected
**Given:** the chatbot is fully wired  
**When:** the Node/Express upload → match → send flow is executed  
**Then:** all existing tests continue to pass

## Scope

### IN
- Wiring: `webhook handler (6.1)` → `AI service (6.2)` → `Evolution API send`.
- Rate limiting table or in-memory counter keyed by normalised phone number.
- `CHATBOT_RATE_LIMIT_DAILY` environment variable (default: 10).
- Send reply via Evolution API (`/message/sendText/{instance}` — already used by broadcast).
- Fallback message delivery on AI failure.
- "Number not registered" reply.
- Structured logging for all chatbot interactions.
- End-to-end integration test with mocked Evolution API and mocked AI API.

### OUT
- Persistent conversation history / multi-turn memory.
- Chatbot analytics dashboard.
- Admin interface to view chatbot logs in UI.
- Support for media messages (images, audio, documents).

## Risks

| Risk | Mitigation |
|------|------------|
| Evolution API send fails mid-flow | Log failure per delivery; do not retry indefinitely; student may not receive reply in edge cases |
| Rate limit state lost on restart | Acceptable for MVP (in-memory or lightweight DB counter); document limitation |
| Chatbot interferes with broadcast flow | Chatbot uses a separate router and service; broadcast path unchanged |
| End-to-end test requires live APIs | Mock Evolution API and AI API in tests; document manual smoke test procedure |

## Dev Notes

### Pipeline flow
```python
async def handle_chatbot_message(phone: str, text: str):
    student = await identify_student_by_phone(phone)
    if not student:
        await send_whatsapp(phone, UNKNOWN_NUMBER_MSG)
        return
    if rate_limiter.is_blocked(phone):
        await send_whatsapp(phone, RATE_LIMIT_MSG)
        return
    try:
        response = await ai_chatbot.generate_grade_response(student.id, text)
    except Exception as e:
        log.error("AI failure", error=e, student_id=student.id)
        response = FALLBACK_MSG
    rate_limiter.record(phone)
    await send_whatsapp(phone, response)
    log_chatbot_interaction(phone, student.id, outcome="success")
```
