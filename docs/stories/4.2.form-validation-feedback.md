## Story
**As a** professora que usa a aplicação,
**I want** validação de formulário em tempo real antes do envio,
**so that** receba feedback imediato sobre erros sem ter de esperar pela resposta do servidor, reduzindo uploads inválidos e confusão.

## Acceptance Criteria

### AC-1: Validação de formato e tamanho do ficheiro CSV
- **Given** a professora selecciona um ficheiro no input de estudantes ou de notas,
- **When** o ficheiro não tem extensão `.csv` ou ultrapassa 5 MB,
- **Then** aparece uma mensagem de erro inline imediatamente abaixo do input (não um `alert()`), descrevendo o problema específico: "Apenas ficheiros .csv são aceites." ou "O ficheiro excede o tamanho máximo de 5 MB."

### AC-2: Mensagem de erro inline acessível por ARIA
- **Given** um erro de validação é apresentado para um input de ficheiro,
- **When** a mensagem de erro está visível no DOM,
- **Then** o elemento de erro tem `id` único (ex: `studentsFileError`, `gradesFileError`) e o `<input>` correspondente referencia-o via `aria-describedby`, garantindo que leitores de ecrã anunciem o erro ao focar o input.

### AC-3: Botão de submit desabilitado enquanto campos são inválidos
- **Given** o formulário de upload tem um ficheiro inválido (formato errado, tamanho excedido) ou nenhum ficheiro seleccionado,
- **When** o estado de validação é inválido,
- **Then** o botão de submit correspondente (`Importar estudantes` / `Importar notas`) tem `disabled` aplicado e indicação visual clara (cursor not-allowed, opacidade reduzida), impedindo o envio prematuro.

### AC-4: Botão de submit reactivado quando input é válido
- **Given** a professora corrige o ficheiro (selecciona um `.csv` válido dentro do limite de 5 MB),
- **When** o evento `change` dispara com ficheiro válido,
- **Then** o atributo `disabled` é removido do botão de submit e a mensagem de erro é limpa — sem necessidade de recarregar a página.

### AC-5: Limpeza de erros ao corrigir o input
- **Given** existe uma mensagem de erro inline visível para um input de ficheiro,
- **When** a professora selecciona um novo ficheiro válido,
- **Then** a mensagem de erro desaparece (elemento de erro fica vazio ou `hidden`) antes de qualquer submissão.

### AC-6: Validação do textarea de mensagem (template WhatsApp)
- **Given** a professora tenta disparar em massa (botão "Disparar em massa"),
- **When** o campo `#template` está vazio ou contém apenas espaços em branco,
- **Then** aparece uma mensagem de erro inline ("A mensagem não pode estar vazia.") abaixo do textarea e o botão de disparo não executa o `fetch`.

### AC-7: Validação da confirmação antes do envio real
- **Given** a professora clica "Disparar em massa" com `dryRun` desmarcado e `confirmReal` também desmarcado,
- **When** a validação client-side corre,
- **Then** aparece uma mensagem de erro inline ("Marca a confirmação de envio real ou activa a simulação.") em vez do `alert()` actual — o comportamento de bloqueio mantém-se mas a UX usa feedback inline acessível.

### AC-8: Indicador visual de estado do campo (válido / inválido / neutro)
- **Given** qualquer input de ficheiro ou textarea de mensagem,
- **When** o campo tem erro de validação,
- **Then** recebe a classe CSS `.field-invalid` (border vermelho, ícone ou indicador visual) e, quando válido após interacção, recebe `.field-valid` (border verde/neutro); o estado neutro (sem interacção) não mostra nem válido nem inválido.

### AC-9: Mensagens de erro descritivas em português
- **Given** qualquer erro de validação client-side,
- **When** a mensagem é apresentada,
- **Then** a mensagem é específica e em português: nunca genérica ("Erro") mas descritiva ("Apenas ficheiros .csv são aceites.", "O ficheiro excede o tamanho máximo de 5 MB.", "A mensagem não pode estar vazia.", "Selecciona um ficheiro CSV antes de importar.").

### AC-10: Erros anunciados em regiões aria-live
- **Given** uma mensagem de erro inline é injectada ou removida do DOM,
- **When** o estado de validação muda,
- **Then** as regiões de status existentes (`#studentsStatus`, `#gradesStatus`) com `aria-live="polite"` são actualizadas de forma coerente, OU os novos elementos de erro inline têm `aria-live="assertive"` quando o erro é crítico (ex: formato inválido ao submeter), garantindo que utilizadores com leitor de ecrã sejam notificados sem depender apenas de foco.

## Scope

### IN
- Validação client-side de inputs `#studentsFile` e `#gradesFile` (formato `.csv`, tamanho máx 5 MB)
- Validação do textarea `#template` (não vazio antes do disparo)
- Substituição do `alert()` no handler de `sendBtn` por feedback inline
- Aplicação de `disabled` nos botões de submit quando inválido
- Estilos CSS para `.field-invalid`, `.field-valid` em `public/styles.css`
- Elementos de erro inline no HTML (`<p id="studentsFileError">`, `<p id="gradesFileError">`, `<p id="templateError">`, `<p id="sendConfirmError">`)
- Atributos `aria-describedby` nos inputs ligados aos novos elementos de erro
- Testes unitários/integração para lógica de validação client-side

### OUT
- Validação server-side (já coberta pelo Epic 3 / Story 3.2)
- Validação do conteúdo interno do CSV (cabeçalhos, linhas) — escopo do servidor
- Modais de confirmação (cobertos pela Story 4.6)
- Internacionalização / multi-idioma

## Dev Notes

### Estado actual do formulário (extraído de `public/index.html` e `public/app.js`)

**Formulário de estudantes (`#studentsForm`):**
```html
<input type="file" id="studentsFile" accept=".csv" required aria-describedby="studentsStatus" />
<button type="submit">Importar estudantes</button>
<p id="studentsStatus" aria-live="polite" aria-atomic="true"></p>
```
- `accept=".csv"` filtra o diálogo do OS mas não bloqueia programaticamente outros tipos
- `aria-describedby="studentsStatus"` está ligado ao status geral — será necessário adicionar um elemento de erro específico sem remover o link existente (usar múltiplos IDs: `aria-describedby="studentsFileError studentsStatus"`)

**Formulário de notas (`#gradesForm`):**
- Estrutura idêntica ao de estudantes, com `#gradesFile`, `#gradesStatus`

**Validação actual em `uploadCsv()` (linha 21-23 de `app.js`):**
```js
if (!fileInput.files.length) {
  throw new Error("Selecciona um ficheiro CSV");
}
```
- Validação apenas de "ficheiro seleccionado" — sem verificação de extensão ou tamanho
- Erro propagado via `catch` e escrito em `studentsStatus.textContent` — sem elemento de erro inline dedicado

**Botão de envio em massa (`sendBtn`) — linha 183-185 de `app.js`:**
```js
if (!dryRun && !confirmReal) {
  alert("Marca a confirmação para envio real ou usa simulação.");
  return;
}
```
- Usa `alert()` nativo — deve ser substituído por feedback inline (AC-7)
- Sem validação do `#template` vazio

**Textarea `#template`:**
- Sem validação de conteúdo antes do `fetch` a `/api/send/bulk`
- Não tem `aria-describedby` para erros

### Padrão de implementação sugerido

```js
// Validação inline para inputs de ficheiro
function validateFileInput(inputId, errorId, buttonId) {
  const input = document.getElementById(inputId);
  const errorEl = document.getElementById(errorId);
  const button = document.getElementById(buttonId);
  const file = input.files[0];

  if (!file) {
    showError(errorEl, input, "Selecciona um ficheiro CSV antes de importar.");
    button.disabled = true;
    return false;
  }
  if (!file.name.toLowerCase().endsWith('.csv')) {
    showError(errorEl, input, "Apenas ficheiros .csv são aceites.");
    button.disabled = true;
    return false;
  }
  if (file.size > 5 * 1024 * 1024) {
    showError(errorEl, input, "O ficheiro excede o tamanho máximo de 5 MB.");
    button.disabled = true;
    return false;
  }
  clearError(errorEl, input);
  button.disabled = false;
  return true;
}
```

### Decisions
1. Submit buttons given explicit IDs (`studentsSubmitBtn`, `gradesSubmitBtn`) to allow DOM targeting from JS without querying by type/position.
2. `disabled` attribute set directly in HTML (not only via JS) so buttons are unavailable even before `DOMContentLoaded` fires.
3. Validation utilities (`showError`, `clearError`, `validateFileInput`) defined at module scope before listeners so they can be unit-tested via inline re-implementation in the test file.
4. `aria-describedby` extended to include both the new error element and the existing status element (multiple IDs) to preserve Story 4.1 accessibility without regression.
5. Template and confirm-error elements cleared on each `sendBtn` click before re-validation to avoid stale error state.
