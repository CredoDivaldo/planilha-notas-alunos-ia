# Story 4.7: State Gating

**Epic:** EPIC-4 — UX Refinement
**Status:** InReview
**Points:** 3
**Priority:** P1
**Owner:** @ux-design-expert + @dev
**Executor:** @dev
**Quality Gate:** @qa

---

## Description

Como utilizador do MVP Notas + WhatsApp, quero que os passos do workflow sejam bloqueados ou desbloqueados de forma dinâmica conforme os pré-requisitos forem cumpridos, para que eu não possa executar acções fora de ordem e receba orientação clara sobre o que fazer a seguir.

O fluxo obrigatório é:
1. **Criar instância** (Secção 4) → desbloqueia "Ligar WhatsApp"
2. **Ligar WhatsApp** (estado "open") → desbloqueia "Importar Notas" (Secções 1, 2) e "Gerar Match" (Secção 3)
3. **Importar Notas** → desbloqueia "Disparar em massa" (Secção 5)

---

## Acceptance Criteria

- **AC-1:** A secção "1) Upload de estudantes" e "2) Upload de notas" apenas ficam activas (formulários e botões habilitados) quando a ligação WhatsApp estiver no estado "open" (instância criada E conectada).
- **AC-2:** A secção "5) Mensagem e disparo" (botão "Disparar em massa") apenas fica activa quando a ligação WhatsApp estiver "open" E as notas tiverem sido importadas com match gerado (pelo menos um resultado na tabela de match).
- **AC-3:** Passos bloqueados mostram indicador visual claro: opacidade reduzida (`.section-locked`) e um label/tooltip descritivo explicando o pré-requisito em falta.
- **AC-4:** Passos desbloqueados mostram indicador visual de disponível (`.section-unlocked`), sem opacidade reduzida.
- **AC-5:** A ordem do workflow é respeitada: Criar Instância → Ligar WhatsApp → Importar Notas/Match → Disparar em massa.
- **AC-6:** O estado persiste durante a sessão via `appState` em memória — o polling não repõe o estado a false; só avança (false → true, nunca true → false em operação normal).
- **AC-7:** Acessibilidade: secções bloqueadas têm `aria-disabled="true"` nas suas acções interactivas e `aria-describedby` apontando para uma mensagem que explica o motivo do bloqueio, anunciável por screen readers.

---

## Scope

### IN
- Estado da aplicação: `appState = { instanceCreated: false, connected: false, matchGenerated: false }`
- Função `updateGating()` chamada após cada transição de estado
- CSS `.section-locked` (opacity, cursor) e `.section-unlocked`
- Integração com `createInstanceBtn` handler → `appState.instanceCreated = true`
- Integração com `stopPolling("connected")` → `appState.connected = true`
- Integração com `matchBtn` handler → `appState.matchGenerated = true`
- ARIA: `aria-disabled`, `aria-describedby` em elementos bloqueados
- Testes unitários para `updateGating()` e transições de estado

### OUT
- Persistência entre sessões (localStorage/sessionStorage)
- Animações de transição (fora de scope UX-007)
- Gating da Secção 4 interna (botões "Ligar" e "Ver estado" permanecem sempre visíveis)

## Dev Notes

### Estrutura actual das secções (`public/index.html`)

```
<section aria-labelledby="sec1-heading">  — Upload estudantes (CSV)
<section aria-labelledby="sec2-heading">  — Upload notas (CSV)
<section aria-labelledby="sec3-heading">  — Gerar match (matchBtn)
<section aria-labelledby="sec4-heading">  — Conectar WhatsApp (createInstanceBtn, connectInstanceBtn, stateInstanceBtn)
<section aria-labelledby="sec5-heading">  — Mensagem e disparo (sendBtn)
```

### Controlo de estado actual (`public/app.js`)

- `createInstanceBtn` listener: chama `callEvolution("/api/evolution/instance/create", "POST")` → inicia polling
- `stopPolling("connected")`: chamada quando `state === "open"` — ponto de integração para `appState.connected = true`
- `matchBtn` listener: chama `/api/match/generate` → preenche `matchTableBody` — ponto de integração para `appState.matchGenerated = true`
- Botões de submit (`studentsSubmitBtn`, `gradesSubmitBtn`) já têm `disabled` por defecto — o gating irá manter/remover esse `disabled`
- `sendBtn` não tem `disabled` por defecto — o gating irá adicioná-lo até pré-requisitos cumpridos

### Estratégia de gating

```javascript
const appState = { instanceCreated: false, connected: false, matchGenerated: false };

function updateGating() {
  // Secções 1 e 2 — require connected
  // Secção 3 (match) — require connected
  // Secção 5 (send) — require connected + matchGenerated
  // Secção 4 — sempre activa (ponto de entrada do workflow)
}
```

### ARIA pattern para secção bloqueada
```html
<section aria-labelledby="sec1-heading" aria-describedby="sec1-lock-reason">
  <p id="sec1-lock-reason" class="lock-reason visually-hidden">
    Ligue o WhatsApp primeiro (Secção 4).
  </p>
  <!-- conteúdo da secção -->
</section>
```

## Dev Agent Record

**Decisions:**
- `appState` placed at the top of `app.js` as a module-level variable (session-scoped, no persistence per AC-6)
- `updateGating()` is idempotent — safe to call multiple times; state only advances forward
- Sec4 (WhatsApp setup) intentionally excluded from gating — it is the entry point of the workflow
- `matchGenerated` requires `payload.matched.length > 0` (at least one match), not just API success
- `pointer-events: none` on `.section-locked` prevents all mouse interaction without needing to disable every child element individually
- ESLint config extended for DOM tests to fix pre-existing `confirmation-modal.test.js` lint errors alongside new `state-gating.test.js`
