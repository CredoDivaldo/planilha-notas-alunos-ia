## Story Statement

**As a** operador (Credo),
**I want** subir a stack Evolution local (Redis + Postgres + Evolution API) e enviar uma mensagem de teste WhatsApp real,
**so that** eu prove que o loop Evolution Webhook → FastAPI (`POST /api/v1/chatbot/webhook`) → `EvolutionApiClient` + DeepSeek Chat → resposta real acontece.

---

## Acceptance Criteria

- [ ] **AC1:** `docker compose -f docker-compose.evolution.yml up` levanta `evolution_api`, `evolution_redis`, `evolution_postgres` healthy em `localhost`.
- [ ] **AC2:** Root `.env` contém `EVOLUTION_BASE_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`, `CHATBOT_WEBHOOK_TOKEN`, **`DEEPSEEK_API_KEY`**, `AI_PROVIDER=deepseek`, `AI_MODEL=deepseek-chat`. (`.env.evolution` mantém-se independente da IA — IA vive no FastAPI, não na Evolution.)
- [ ] **AC3:** Evolution webhook está configurado para apontar para `http://localhost:8000/api/v1/chatbot/webhook`.
- [ ] **AC4:** Número WhatsApp de teste pareado com QR (step humano one-time) está operacional.
- [ ] **AC5:** Ao enviar mensagem de um telefone (nº A) → o sistema recebe no FastAPI, chama DeepSeek para gerar resposta, e envia resposta via Evolution → aparece no telefone B (segundo nº com consentimento).
- [ ] **AC6:** `docs/qa/evolution-acceptance-9.1.md` é actualizado com transcript completo (timestamps, payloads relevantes, resposta gerada, evidência do envio WhatsApp).

## Risks

| # | Risk | Likelihood | Mitigation |
|---|------|------------|------------|
| R1 | **Docker daemon indisponível** no host (Docker Desktop não está a correr, ou socket não existe em `~/.docker/run/docker.sock`) | Medium (verificado em 2026-06-08 ~21:00 que daemon estava down; recuperado com `open -a Docker`) | (a) Antes de T1, validar `docker ps` 1x; (b) se falhar, tentar `open -a Docker` + aguardar 30-60s; (c) se falhar, documentar em `docs/qa/evolution-acceptance-9.1.md` como blocker e marcar T1 como BLOCKED (não FAIL — story não pode ser "Done" mas pode ser "Partial complete"); (d) operador pode correr os comandos manualmente e anexar output ao transcript |
| R2 | **QR pairing window expira** (Evolution QR tem TTL curto, tipicamente 60-120s) | High se o operador não estiver com o telemóvel B pronto antes de T4 | (a) T4 step tem de ser executado num único bloco atómico: criar/refresh instance → imediatamente pedir QR via `GET /instance/connect/{name}` → operador scaneia dentro de 60s; (b) script helper `docs/runbooks/evolution.md` (a entregar em Story 9.3) deve sequenciar isto; (c) fallback: usar `pairingCode` se Evolution API v2.x o expor (mais robusto que QR) |
| R3 | **Network egress bloqueado** para `api.deepseek.com` (firewall corporativo, proxy institucional, ou rate-limit) | Low–Medium dependendo do ambiente | (a) Antes de T5, validar com `curl -sSI https://api.deepseek.com/health` (deve responder 200/301); (b) FastAPI logs mostrarão `deepseek_api_call failed` com HTTP status se bloqueado; (c) fallback documentado: Evolution dry-run path retorna 200 com `ai_called: false` (mensagem de fallback) — aceitável para validar o **wiring** mas **não** para validar AC5 real |
| R4 | **WhatsApp delivery latency / 3rd-party flakiness** (Evolution → WhatsApp Business protocol pode ter delays de 1-30s, ou rate-limits internos do WhatsApp) | Medium em janela de carga alta; Low em idle | (a) Aceitar latency até 60s antes de marcar T5 como falha; (b) confirmar via `docker logs evolution_api` que `sendMessage` foi dispatched (não só accepted); (c) operator attestation no telefone B é a evidência definitiva; (d) se 1ª tentativa falha, retry 2x com backoff exponencial antes de classificar como blocker |

### Risks adicionais monitorizados (não bloqueantes)

- **R5 (baixa)**: `EVOLUTION_API_KEY` no `.env` é o valor default do `.env.evolution` (`02fb2b6a...`). Em produção deveria ser regenerado. **Mitigação:** Story 9.3 (runbook) documenta rotação; Story 9.1 apenas verifica que wiring funciona.
- **R6 (baixa)**: `AI_MODEL=deepseek-chat` é hard-coded no `.env.example`. Se DeepSeek depreciar este model, `ai_chatbot.py` factory vai logar warning. **Mitigação:** monitorar release notes DeepSeek; troca é 1 linha de `.env`.
