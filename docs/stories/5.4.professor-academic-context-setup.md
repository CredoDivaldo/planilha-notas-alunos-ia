## Story

**As a** professor,  
**I want** to configure my own academic contexts (turma + subject + semester + shift),  
**so that** I can manage the correct class, subject, semester and shift before importing grades or publishing information.

---

## Acceptance Criteria

### AC-1: Academic context model is defined
**Given:** the system requires explicit scoping for professor grade management  
**When:** the academic context foundation is implemented  
**Then:** the data model defines `academic_contexts` table with fields: `turma` (class/group), `subject`, `semester`, `shift`, `professor_id`, `academic_year`, `status` and immutable `created_at`

### AC-2: Professor can have multiple active contexts
**Given:** a professor may teach the same subject in multiple turmas  
**When:** contexts are assigned to a professor  
**Then:** a professor can have multiple simultaneous active contexts without collision, and contexts are isolated by `(professor_id, turma, subject, semester, shift)` compound uniqueness

### AC-3: Same subject exists in multiple contexts without collision
**Given:** a subject may appear in different turmas and shifts  
**When:** grades are entered under a context  
**Then:** the same subject in turma A, shift 1 is distinct from the same subject in turma B, shift 2, and each context maintains independent grades

### AC-4: Class rosters are scoped to context
**Given:** each academic context requires student membership  
**When:** context setup is completed  
**Then:** `class_enrollments` table links students to contexts via `(academic_context_id, student_id)` with enrollment status `active`, `dropped`, `completed`

### AC-5: Subject configuration within context
**Given:** grade entry and calculation require subject-specific rules  
**When:** a context is created  
**Then:** `context_subject_configuration` stores configurable fields: assessment count, weight distribution, calculation formula version, and derived state rules per context

### AC-6: Semester and shift hierarchy is established
**Given:** academic structure organizes into semesters and shifts  
**When:** the model is finalized  
**Then:** `semesters` and `shifts` are master reference tables, each context references exactly one semester and one shift, and semesters have `start_date`, `end_date` and `academic_year`

### AC-7: Context lifecycle is tracked
**Given:** professor needs visibility into context status  
**When:** contexts transition states  
**Then:** contexts track lifecycle state: `draft`, `active`, `archived`, with audit events for creation, activation, and archival

### AC-8: Uploads are scoped to a context
**Given:** grade imports must be placed in a specific academic context  
**When:** a professor initiates an upload  
**Then:** uploads are required to specify an `academic_context_id`, and the grade entry import is validated against that context's student roster and subject configuration

## Risks

| Risk | Mitigation |
|------|------------|
| Professor creates overlapping contexts | Enforce `(professor_id, turma, subject, semester, shift)` uniqueness at DB level |
| Student enrolled in multiple contexts unintentionally | Roster API requires explicit context ID; audit trail captures assignment |
| Context configuration is too rigid for formula variance | Store `context_subject_configuration` as flexible JSON blob; avoid hard-coded calculation rules |
| Bulk context creation is manual and slow | Provide CLI import for bulk turma/subject/semester/shift setup from CSV or legacy JSON |
| Grade scoping breaks existing JSON upload flow | Preserve JSON import with context inference rules; allow professor to map uploads to contexts during transition |

## Dev Notes

### Academic context as the primary scoping boundary
All grade entry, calculation, publication and student read operations are scoped to a specific `academic_context_id`. This ensures that a professor's multiple turmas and subjects do not collide in the data model or in query results.

### Semester and shift as reference data
`semesters` and `shifts` are controlled by the institution and populated during bootstrap. Stories do not create semesters/shifts on the fly; they reference existing records.

### Subject configuration per context
The same subject in different contexts may have different assessment counts, weights, or derived-state rules. Store this in `context_subject_configuration` as flexible JSON; do not hard-code formulas in application logic.

### CSV bulk import for local MVP
Professors may need to create multiple contexts. Provide a CLI command for CSV bulk import to speed local/development setup without requiring UI.

### Transition rule: JSON uploads gain context scope
Current JSON upload flow does not specify context. During transition, implement a professor-facing mapping flow: "Select the context for this upload" before processing. Log this decision in `grade_entries.source_context` or similar audit field.
