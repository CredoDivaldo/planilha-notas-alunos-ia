## Story Statement

**As a** operador (Credo),
**I want** substituir o provedor de IA Baidu QianFan pelo DeepSeek Chat em toda a pipeline de chatbot,
**so that** o Epic 9 (acceptance run Evolution) possa ser executado contra um provedor de IA com chave de API já disponível, sem a fricção operacional de obter conta QianFan free-tier.

---

## Background

O Epic 6 (2026-06-02) fixou Baidu QianFan como provedor default com a justificativa de "free tier, sem custos". Em 2026-06-08 Credo obteve acesso a uma chave de API DeepSeek e decidiu trocar o provedor **antes** do Epic 9 arrancar. O switch foi analisado via `*correct-course` e formalizado no Sprint Change Proposal em `docs/qa/change-proposals/2026-06-08-deepseek-switch.md`.

## Acceptance Criteria

### AC-1: New `DeepSeekProvider` is implemented
**Given:** o sistema precisa de um provider DeepSeek que respeite a interface `AIProvider`  
**When:** `AIGradeQueryService` é instanciado com `AI_PROVIDER=deepseek`  
**Then:** um `DeepSeekProvider` é construído e a chamada `call(system_prompt, user_message)` retorna resposta de `https://api.deepseek.com` via SDK OpenAI-compat

### AC-2: BaiduProvider is removed
**Given:** o switch é permanente (não há fallback para Baidu)  
**When:** o sistema arranca  
**Then:** `baidu_provider.py` não existe e `BaiduProvider` não é importável

### AC-3: `qianfan` dependency is removed
**Given:** o switch elimina o uso do SDK QianFan  
**When:** `pip install` corre com o `pyproject.toml` actualizado  
**Then:** nenhuma referência a `qianfan` ou `BAIDU_API_KEY` permanece em `pyproject.toml` nem em runtime env

### AC-4: `.env.example` reflects DeepSeek as default
**Given:** novos contribuidores devem arrancar com DeepSeek por default  
**When:** alguém copia `.env.example` para `.env`  
**Then:** encontra `AI_PROVIDER=deepseek`, `AI_MODEL=deepseek-chat` e referência a `DEEPSEEK_API_KEY` (já presente na secção LLM Providers, linha 15)

### AC-5: `AIGradeQueryService` factory keeps legacy providers
**Given:** Claude e OpenAI continuam a ser fallbacks úteis para dev/test  
**When:** `AI_PROVIDER=claude` ou `AI_PROVIDER=openai` está configurado  
**Then:** o serviço instancia o provider legacy correspondente (lê `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` directamente, com fallback a `AI_API_KEY` legacy)

### AC-6: Test bootstrap uses DeepSeek
**Given:** o default de produção é DeepSeek, os testes devem reflectir isso  
**When:** `pytest` corre sem envs manuais  
**Then:** `conftest.py` define `AI_PROVIDER=deepseek` e `DEEPSEEK_API_KEY=test-deepseek-api-key-for-testing` por default

### AC-7: Existing tests still pass
**Given:** os testes existentes usam `AI_PROVIDER=claude`+mock (não dependem do provider real)  
**When:** `pytest backend/tests/test_ai_chatbot.py` corre após o switch  
**Then:** todos os testes verdes (a interface `AIProvider` é abstraída; o mock de `service.provider.call` continua a funcionar)

### AC-8: Documentation reflects the switch
**Given:** auditoria futura precisa de rastreabilidade  
**When:** alguém lê o codebase em busca de "Baidu"  
**Then:** apenas referências históricas aparecem (marcadas como "DEPRECATED 2026-06-08" ou "Revised 2026-06-08"), e a tabela em `docs/architecture/system-architecture.md` lista DeepSeek como provider actual

## Dev Notes

### DeepSeek API quick reference

- Endpoint: `https://api.deepseek.com` (OpenAI-compatible)
- Auth: `Authorization: Bearer $DEEPSEEK_API_KEY`
- Model: `deepseek-chat` (default)
- Custo: ~$0.14/M input tokens, ~$0.28/M output tokens (significativamente mais barato que GPT-4o)
- Limite: 60 requests/min (tier gratuito); suficiente para 1 turma piloto

### SDK reuse

O pacote `openai>=1.42` já está em `pyproject.toml`. Como DeepSeek expõe API OpenAI-compatible, basta:

```python
from openai import OpenAI
client = OpenAI(api_key=..., base_url="https://api.deepseek.com")
client.chat.completions.create(model="deepseek-chat", ...)
```

**Zero dependências novas.** O `qianfan` sai limpo.

### Test strategy

- Não adicionar testes E2E para DeepSeek (requer chave real; coberto em Story 9.1)
- Manter `pytest backend/tests/test_ai_chatbot.py` verdes — usa mock via `patch.object(service.provider, "call", ...)` que abstrai o provider
- O `test_claude_provider_requires_api_key` e `test_openai_provider_requires_api_key` continuam a ser cobertura válida para os legacy fallbacks

## Risks

| Risk | Mitigation |
|------|-----------|
| DeepSeek outage durante Epic 9 acceptance | O `AIProvider` ABC permite fallback rápido para `claude` ou `openai` (basta trocar `AI_PROVIDER`) |
| Custo de tokens maior que o esperado | DeepSeek é $0.14/M (muito abaixo de GPT-4o ~$5/M); free tier tem rate limit mas é generoso para 1 turma |
| Latência superior a Baidu | 200-500ms (vs Baidu 150-300ms); aceitável para fluxo assíncrono WhatsApp |
| Mudança quebra a interface `AIProvider` | A interface é só `call(system_prompt, user_message) -> str` — DeepSeek respeita. Zero risco |
