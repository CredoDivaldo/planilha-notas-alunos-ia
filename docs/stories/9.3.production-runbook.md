## Acceptance Criteria

- [x] **AC1:** `docs/runbooks/evolution.md` criado com instruções de bring-up da stack (Redis + Postgres + Evolution API), incluindo `docker compose -f docker-compose.evolution.yml up -d` e variáveis de ambiente obrigatórias (`EVOLUTION_BASE_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE`, `CHATBOT_WEBHOOK_TOKEN`, `DEEPSEEK_API_KEY`, `AI_PROVIDER=deepseek`, `AI_MODEL=deepseek-chat`).
- [x] **AC2:** Secção "Pairing WhatsApp" explica passo-a-passo como parear novo número via Manager UI: `docker compose -f docker-compose.evolution.yml up -d` → abrir `http://localhost:8080/manager` → criar instance (ex: `turma-c`) → ler QR via `GET /instance/connect/{instance}` ou UI → scan com WhatsApp.
- [x] **AC3:** Secção "Key Rotation" cobre rotação de `EVOLUTION_API_KEY` (regenerar no Manager UI → actualizar `.env` → restart `evolution_api` container) e `CHATBOT_WEBHOOK_TOKEN` (gerar novo no `.env` → restart FastAPI → validar via `curl POST /api/v1/chatbot/webhook` com novo token).
- [x] **AC4:** Secção "Failure Modes" cobre 5 cenários com sinais observáveis:
  - Evolution 5xx (502 do FastAPI, container unhealthy)
  - DeepSeek 429 (chatbot_pipeline retorna "limite atingido", `user_message` da error)
  - Postgres locked / disk full (Evolution API travada, `pg_isready` falha)
  - Docker kill mid-broadcast (sanitised 502 — comportamento documentado em Story 9.2)
  - Evolution instance disconnected (status `disconnected` no Manager UI)
- [x] **AC5:** Secção "Log Locations & Grep Recipes" lista:
  - FastAPI logs: `docker compose logs fastapi -f | grep chatbot` (ou `journalctl` em produção)
  - Evolution logs: `docker logs evolution_api -f 2>&1 | grep -i error`
  - Postgres logs: `docker logs evolution_postgres -f 2>&1 | grep -E "FATAL|ERROR"`
  - Specific grep recipes: webhook hits, evolution_api errors, rate-limit hits, sanitisation leaks
- [x] **AC6:** Secção "Backup/Restore Postgres" cobre:
  - Backup: `docker exec evolution_postgres pg_dump -U user evolution > backup-$(date +%F).sql`
  - Restore: `cat backup-YYYY-MM-DD.sql | docker exec -i evolution_postgres psql -U user evolution`
  - Verificação: `docker exec evolution_postgres psql -U user -d evolution -c "SELECT count(*) FROM instances;"`
  - Cron-friendly: atalhos, retenção sugerida (7 daily + 4 weekly)
- [x] **AC7:** Runbook inclui secção "Sanitisation Contract" (Story 9.2 hardening) com a lista de 10 patterns que `EvolutionApiError.user_message` strip — qualquer operator que vê um 502 deve poder confirmar se a mensagem foi sanitizada.
- [x] **AC8:** Runbook inclui secção "Rate Limiting" (Story 3.4) com default 10/dia, como ajustar, e onde ver o contador em runtime (`ChatbotRateLimiter._counters`).
- [x] **AC9:** Secção "Architecture Reference" aponta para `docs/architecture/` (Epic 6 + Epic 8 sections) e `docs/qa/evolution-acceptance-9.1.md` (E2E transcript) para contexto.
- [x] **AC10:** Runbook é runnable end-to-end: o conteúdo cobre os 6 tópicos do scope IN do Epic 9, e um operador (Credo ou sucessor) consegue seguir sem ajuda externa.
