## Story

**As a** student,  
**I want** to see my published exam, recurso, and prova dates in the portal calendar,  
**so that** I know when my assessments are scheduled without depending solely on broadcast messages.

---

## Acceptance Criteria

### AC-1: Calendar table created
**Given:** `published_calendar_snapshots` table does not exist  
**When:** Alembic migration is applied  
**Then:** `published_calendar_snapshots` table exists with columns: `id`, `academic_context_id` (FK→academic_contexts), `student_id` (FK→students, nullable for class-wide events), `event_type` (exam|recurso|prova), `subject`, `start_date`, `end_date` (nullable), `location` (nullable), `is_published` BOOLEAN DEFAULT False, `published_at` (nullable), `broadcast_job_id` (FK→broadcast_jobs, nullable), `created_at`, `updated_at`

### AC-2: Calendar query reads only published snapshots
**Given:** calendar events may be draft (`is_published=False`) or published (`is_published=True`)  
**When:** student queries `GET /api/v1/portal/me/calendar`  
**Then:** only events WHERE `is_published=True` are returned; draft events are excluded

### AC-3: Calendar scoped to authenticated student
**Given:** a student is enrolled in specific academic contexts  
**When:** calendar is requested  
**Then:** only events linked to the student's enrolled contexts (via `academic_context_id`) are returned

### AC-4: Calendar stub replaced with real implementation
**Given:** `PortalService.get_calendar()` currently returns a hardcoded empty list with a TODO comment  
**When:** this story is implemented  
**Then:** `get_calendar()` executes a real query against `published_calendar_snapshots` and returns published events for the authenticated student

### AC-5: Existing portal tests continue to pass
**Given:** Story 5.5 has 78 passing tests  
**When:** this story is implemented  
**Then:** all 78 existing tests still pass, plus new calendar tests are added

## Dev Notes

### Calendar query rule
Only events with `is_published=True` and linked to the authenticated student's enrolled `academic_context_ids` (resolved via `ClassEnrollment`) are returned. No draft calendar data is ever exposed through the portal.

### Implementation guidance
- The `get_calendar()` stub in `portal/service.py` returns `[]` with a TODO comment referencing the missing table. Replace it with a SELECT that joins `published_calendar_snapshots` against the student's enrolled `academic_context_id` values.
- `student_id` on the snapshot is nullable to support class-wide events; the query should return rows where `student_id IS NULL OR student_id = :authenticated_student_id`.
- Pydantic response model for calendar events must use `extra="forbid"` consistent with other portal models.

### QA issue resolved
This story closes QA issue `5.5-001` (medium severity, REQ category) from the Story 5.5 gate: "AC-4 calendar service is a hardcoded empty-list stub; published_calendar_snapshots table and query not yet implemented."
