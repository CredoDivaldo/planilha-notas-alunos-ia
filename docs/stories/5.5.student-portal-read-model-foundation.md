## Story

**As a** student,  
**I want** to consult my published academic information in one portal,  
**so that** I can see current grades, academic state and calendar without depending only on broadcast messages.

---

## Acceptance Criteria

### AC-1: Portal reads only published snapshots
**Given:** internal grades and calendar drafts are not student-visible  
**When:** a student opens portal data  
**Then:** portal endpoints read only current `publication_snapshots` and `published_calendar_snapshots`

### AC-2: Student identity scopes all reads
**Given:** students authenticate with student number and password  
**When:** portal endpoints are queried  
**Then:** the backend uses the authenticated student identity to scope records and does not accept arbitrary student-number override for another student

### AC-3: Portal aggregates academic summary
**Given:** a student may have several disciplines in the same semester  
**When:** portal summary is requested  
**Then:** the read model can return turma, curso, semester/context labels, published grades and published academic state per discipline

### AC-4: Calendar is publication-gated
**Given:** exam, recurso and prova dates may change internally  
**When:** calendar is shown to the student  
**Then:** only published calendar snapshots are visible, and draft/internal calendar events are excluded

### AC-5: Internal history is hidden
**Given:** the portal must show the current version only  
**When:** a student queries grades or calendar  
**Then:** detailed audit history, previous snapshots, internal notes and draft calculation payloads are not exposed

### AC-6: Formula uncertainty is visible only as supported state
**Given:** official formula and derived statuses are pending product validation  
**When:** the portal displays published academic state  
**Then:** it displays only state values already present in the published snapshot payload, without recalculating or inventing final institutional labels

## Risks

| Risk | Mitigation |
|------|------------|
| Portal leaks internal/draft data | Query only snapshot tables; add negative tests against internal tables |
| Student accesses another student's data | Scope by authenticated session identity and ignore external student override |
| Calendar appears before publication | Read only `published_calendar_snapshots` |
| Open formula creates inconsistent display | Render only published snapshot payload and version metadata |

## Dev Notes

### Portal source rule
The portal is a read-only consumer of published snapshots. It must not compute grades from internal records and must not expose draft calendar events.

### Product decisions isolated
- Formula/status: portal renders published values as stored.
- Password delivery: handled by Story 5.2, not by portal read model.
- Delegate approval: delegate can access own student portal; technical delegate permissions remain outside portal read endpoints.
