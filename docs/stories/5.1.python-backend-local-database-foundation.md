## Story

**As a** development team,  
**I want** the system foundation to move to a FastAPI backend with a local SQLite database,  
**so that** the academic platform can evolve on a domain model that supports the new product requirements without breaking the current MVP flow.

---

## Acceptance Criteria

### AC-1: FastAPI backend foundation exists
**Given:** the architecture defines Python 3.12+ and FastAPI as the principal backend target  
**When:** the backend foundation is created  
**Then:** the project has `backend/app/main.py`, a Python dependency manifest, and local execution documentation for `python -m uvicorn backend.app.main:app --reload --port 8000`

### AC-2: SQLite and Alembic foundation exists
**Given:** `data/app.sqlite3` is the target local relational database  
**When:** persistence foundation is implemented  
**Then:** SQLite is initialised through Alembic migrations under `backend/migrations/`, with WAL enabled at runtime where applicable

### AC-3: Core academic schema is represented
**Given:** the product depends on academic domain modelling  
**When:** the first schema version is created  
**Then:** it includes at minimum `users`, `students`, `professors`, `courses`, `semesters`, `shifts`, `class_groups`, `subjects`, `teaching_assignments`, `enrollments`, `assessment_definitions`, `grade_entries`, `calculation_results`, `broadcast_jobs`, `notification_deliveries`, `publication_snapshots`, `calendar_events`, `published_calendar_snapshots` and `audit_log`

### AC-4: Legacy JSON is isolated
**Given:** existing JSON files remain part of the MVP transition  
**When:** the Python backend is introduced  
**Then:** JSON files in `data/` are treated as legacy/coexistent import sources, are not deleted or rewritten by migrations, and are not the source of truth for new academic features

### AC-5: API namespace and documentation path are established
**Given:** new API endpoints must coexist with Node legacy endpoints  
**When:** the FastAPI app is bootstrapped  
**Then:** new endpoints are namespaced under `/api/v1`, OpenAPI remains generated by FastAPI, and export to `docs/api/openapi.json` is documented as a follow-up once endpoints exist

### AC-6: Quality and run instructions are updated
**Given:** the project will include Python runtime in addition to Node  
**When:** foundation work is completed  
**Then:** local run commands, Python quality commands, and the mandatory Node regression gates are documented

## Dev Notes

### Architecture decisions already closed
- Python backend: FastAPI.
- Entry point: `backend/app/main.py`.
- API namespace: `/api/v1`.
- Database: SQLite at `data/app.sqlite3`.
- Migrations: Alembic in `backend/migrations/`.
- JSON: legacy/coexistent import source only.
- Node/Express endpoints remain available until parity is validated.

### Source-backed implementation context
- FastAPI entrypoint, SQLite path, WAL, Alembic, JSON coexistence, `/api/v1` and quality command decisions are closed architecture decisions. [Source: docs/architecture/decises-arquitecturais-resolvidas.md]
- The target stack is Python 3.12+, FastAPI, Uvicorn, SQLite, SQLAlchemy 2.x, Alembic, Pydantic 2.x, pytest, Ruff and mypy. [Source: docs/architecture/alinhamento-de-tech-stack.md]
- The initial schema must be born through Alembic v1 and include the academic tables, indexes and rollback/import strategy described in the data model document. [Source: docs/architecture/modelos-de-dados-e-estratgia-de-esquema.md]
- New endpoints use `/api/v1`; FastAPI-generated OpenAPI should be exported to `docs/api/openapi.json` when contracts stabilise. [Source: docs/architecture/estratgia-de-api-e-integrao.md]
- Python modules belong under `backend/app/`, migrations under `backend/migrations/`, tests under `backend/tests/`, and `pyproject.toml` at repository root. [Source: docs/architecture/integrao-com-source-tree.md]
- Python application code should use type hints, Pydantic external contracts, SQLAlchemy models/repositories, Alembic migrations, absolute imports from `backend.app`, uniform API errors and no sensitive logging. [Source: docs/architecture/normas-de-cdigo-e-convenes.md]

### Product decisions not closed in this story
- Official formula and derived academic statuses.
- Exact professor validation flow for delegate-assisted actions.
- Exact channel and ceremony for initial student password delivery.

Implement configurable placeholders and audit hooks where these decisions touch the schema; do not encode institutional rules as final.

### Project Structure Notes
- Add `backend/` in parallel with the existing Node/Express app; do not route existing Node code through Python or make Python import JavaScript modules.
- Keep `src/`, `public/`, `tests/` and existing `data/*.json` files operational during the transition.
- `data/app.sqlite3` is the target academic database only for features migrated by story scope; coexistence must document which flow reads each source.
