## Story

**As a** system administrator,
**I want** all API routes to require an API key,
**so that** the backend is not publicly accessible without credentials and accidental or malicious access is prevented.

---

## Acceptance Criteria

### AC-1: Requests without API key are rejected
**Given:** `API_KEY` is configured in `.env`
**When:** a request to any `/api/*` route is made without the `X-API-Key` header
**Then:** the server returns HTTP 401 with `{ "error": "Não autorizado" }`

### AC-2: Requests with an incorrect API key are rejected
**Given:** `API_KEY` is set in `.env`
**When:** a request includes `X-API-Key` with an incorrect value
**Then:** the server returns HTTP 401 with `{ "error": "Não autorizado" }` — no information about whether the key exists or not

### AC-3: Requests with the correct API key are accepted
**Given:** `API_KEY` is set in `.env` and the client sends the correct value
**When:** the request reaches any `/api/*` route
**Then:** the request is processed normally with the expected response

### AC-4: Static files remain publicly accessible
**Given:** the auth middleware is applied only to `/api/*`
**When:** a request is made to a static file (e.g., `GET /`, `GET /index.html`)
**Then:** the file is served without requiring any API key

### AC-5: Frontend includes API key in all API calls
**Given:** the auth middleware is active
**When:** the static frontend (`public/`) makes any API call
**Then:** it includes the configured `API_KEY` in the `X-API-Key` header so the UI continues to function

### AC-6: Missing `API_KEY` env var causes startup failure
**Given:** `API_KEY` is not set or is empty in `.env`
**When:** `node src/server.js` is executed
**Then:** startup validation (Story 3.1) catches it and the process exits with code 1

## Risks

| Risk | Mitigation |
|------|------------|
| Auth middleware breaks all existing Supertest tests (they don't send the key) | Set `API_KEY` in Jest test environment via `jest.config.js` or `process.env.API_KEY = 'test-key'` in test setup; include key in `supertest` requests |
| Frontend JS files need updating — risk of breaking UI | Review all `fetch` calls in `public/` and add header to each; test manually |
| Timing attack on key comparison | Use `crypto.timingSafeEqual` for the comparison to prevent timing-based key enumeration |

## Dev Notes

### Timing-safe comparison
```javascript
const crypto = require('crypto');

function authMiddleware(req, res, next) {
  const key = req.headers['x-api-key'] || '';
  const expected = process.env.API_KEY || '';
  if (
    key.length !== expected.length ||
    !crypto.timingSafeEqual(Buffer.from(key), Buffer.from(expected))
  ) {
    return res.status(401).json({ error: 'Não autorizado' });
  }
  next();
}
```

### Frontend API key exposure
For the static MVP frontend, the simplest approach is a `window.__API_KEY__` variable injected into `public/index.html` via a `<script>` tag, reading from a config endpoint or hardcoded in a separate `public/config.js` not committed to git. Document the chosen approach in Dev Notes on completion.

### Test setup
Add to `jest.config.js`:
```javascript
testEnvironment: 'node',
globalSetup: './tests/setup.js', // sets process.env.API_KEY
```
Or per-file: `process.env.API_KEY = 'test-key';` at top of each test.
