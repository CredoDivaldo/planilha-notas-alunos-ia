## Story

**As a** product owner,  
**I want** an authentication and role foundation for professor, delegate and student,  
**so that** access to notes and operational features is controlled from the start.

---

## Acceptance Criteria

### AC-1: Student login model is defined
**Given:** students access the portal with student number and password  
**When:** authentication foundation is implemented  
**Then:** the data model supports student-number login, Argon2id password hashing and first-access mandatory password change through `must_change_password=true`

### AC-2: Server-side session foundation exists
**Given:** the architecture defines cookie-based sessions  
**When:** login succeeds  
**Then:** the backend creates a server-side session, returns only an `HttpOnly` cookie with `SameSite=Lax` or stricter where feasible, and rotates the session on login and password change

### AC-3: Professor account model exists
**Given:** professors must access their own operational area  
**When:** the role model is defined  
**Then:** professors have individual accounts distinct from student identities and scoped to their own academic contexts

### AC-4: Delegate role is scoped and not final-authoritative
**Given:** delegates are students with limited technical access  
**When:** delegate permissions are modelled  
**Then:** delegate access is represented as scoped assignment to turma/context, not as global professor rights, and sensitive actions require a professor-validation state rather than direct execution

### AC-5: Role boundaries are documented
**Given:** destructive and sensitive actions must be restricted  
**When:** role design is completed  
**Then:** allowed, blocked and professor-validation-required actions per role are explicitly documented

### AC-6: Initial password delivery remains product-TBD
**Given:** the exact delivery channel for initial student passwords is not closed  
**When:** credentials are generated  
**Then:** the implementation stores only the hash, marks the password as temporary/one-use, logs delivery intent/status without cleartext secret, and leaves channel/ceremony selection configurable

## Risks

| Risk | Mitigation |
|------|------------|
| Initial passwords leak through logs or UI | Store only hashes; never log cleartext; require first change |
| Delegate receives excessive permissions | Enforce backend authorization by role and context on every use case |
| Sessions are treated as frontend-only security | Use server-side session store and `HttpOnly` cookie |
| Open product decision blocks implementation | Implement auditable placeholders and configurable delivery/approval states |

## Dev Notes

### Required role baseline
- `professor`: manages own contexts, notes, calendar, publication and validation.
- `student`: reads only own published snapshots.
- `delegate`: student with technical assignment limited to turma/context; cannot edit notes directly, delete contexts, or publish without professor validation where sensitive.
- `admin_local`: optional bootstrap role for local setup only, if needed.

### Product decisions isolated
- Delegate flow: implement `requires_professor_validation`, `pending`, `approved`, `rejected` states; do not assume exact UI ceremony.
- Password delivery: implement generated temporary secret and delivery log metadata; do not choose WhatsApp, paper, e-mail or another ceremony as final.
