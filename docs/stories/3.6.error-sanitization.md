## Story

**As a** security-conscious operator,
**I want** all error responses to be sanitized before reaching the client,
**so that** internal implementation details, stack traces, file paths, and sensitive configuration are never exposed in API error responses.

---

## Acceptance Criteria

### AC-1: 5xx responses never include internal error details
**Given:** an unhandled exception or internal error occurs in any route handler
**When:** the global error handler in `src/app.js` responds
**Then:** the response body contains only `{ "error": "Erro interno" }` — no stack trace, no file path, no internal message

### AC-2: 4xx responses include only the sanitized user-facing message
**Given:** a route handler throws a 4xx error (e.g., validation failure, missing file)
**When:** the error reaches the global handler
**Then:** the response body contains `{ "error": "<user-facing message>" }` where the message is the explicitly set `error.message` — no stack trace appended

### AC-3: Errors are logged server-side with full detail
**Given:** any error occurs
**When:** the error handler runs
**Then:** the full error (including stack trace and original message) is logged to `console.error` — the server retains full diagnostic information even though the client does not see it

### AC-4: The `X-Request-Id` header is included in error responses for traceability
**Given:** the global error handler responds with any error status
**When:** the response is sent
**Then:** it includes an `X-Request-Id` header with a short unique identifier linking the client response to the server log entry

## Scope

### IN
- Refactor the global error handler in `src/app.js`

## Dev Notes

### Current error handler (src/app.js)
```javascript
app.use((error, _req, res, _next) => {
  console.error("Erro na API:", error);
  const status = Number(error?.status) || 500;
  const message = status >= 500 ? "Erro interno" : error.message || "Erro na requisição";
  res.status(status).json({ error: message });
});
```
The 5xx branch already returns `"Erro interno"`. The refactor should:
1. Add `req.id` to the log
2. Add `X-Request-Id` to the response
3. Ensure the 4xx path never accidentally leaks internal info beyond `error.message`

### Request ID generation
```javascript
const crypto = require('crypto');
function requestIdMiddleware(req, _res, next) {
  req.id = crypto.randomUUID().slice(0, 8);
  next();
}
```
No external dependency needed.
