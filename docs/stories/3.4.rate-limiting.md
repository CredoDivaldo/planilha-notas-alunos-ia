## Story

**As a** system administrator,
**I want** rate limiting on all API routes,
**so that** the server is protected against abuse and unintentional request floods without blocking normal professor usage.

---

## Acceptance Criteria

### AC-1: Rate limit of 100 requests per minute per IP is enforced
**Given:** `express-rate-limit` middleware is configured
**When:** a single IP address sends more than 100 requests in a 60-second window to any `/api/*` route
**Then:** subsequent requests receive HTTP 429 with a JSON error body `{ "error": "Demasiadas requisições. Tente novamente em breve." }`

### AC-2: Rate limit applies only to `/api/*` routes
**Given:** the rate limiter is configured
**When:** a request is made to a non-API route (e.g., static files under `/`)
**Then:** the rate limiter is not applied and the request is served normally

### AC-3: Rate limit headers are included in responses
**Given:** the rate limiter is active
**When:** any `/api/*` request is made within the limit
**Then:** the response includes `RateLimit-Limit`, `RateLimit-Remaining`, and `RateLimit-Reset` headers (standard `express-rate-limit` headers)

### AC-4: Existing tests are not broken by rate limiter
**Given:** tests use `supertest` which does not share IP state between test runs by default
**When:** `npm run test` is executed
**Then:** all existing tests pass without hitting the rate limit

## Scope

### IN
- Install `express-rate-limit` package
- Configure limiter in `src/app.js` for `/api/*` routes
- JSON error response format consistent with existing error handler

### OUT
- Per-route granular limits (single global limit for all `/api/*` is sufficient)
- Redis-backed distributed rate limiting (in-memory is sufficient for local MVP)
- IP whitelist / bypass for internal requests

## Risks

| Risk | Mitigation |
|------|------------|
| `supertest` in tests may trigger rate limit if many tests run against the same app instance | The in-memory store resets per-process; Jest `--runInBand` shares a single process but requests are few per suite |
| Rate limiter blocks the `/api/health` endpoint | Scope limiter only to `/api/` with exclusions, or accept that health checks are also limited |

## Dev Notes

### Package
`express-rate-limit` v7+ — standard Express middleware, no extra dependencies.

### Recommended config
```javascript
import rateLimit from 'express-rate-limit'; // or require()
const limiter = rateLimit({
  windowMs: 60 * 1000,
  max: 100,
  standardHeaders: 'draft-7',
  legacyHeaders: false,
  handler: (_req, res) => {
    res.status(429).json({ error: 'Demasiadas requisições. Tente novamente em breve.' });
  },
});
app.use('/api/', limiter);
```

### Test isolation
For tests, set `NODE_ENV=test` and configure `skip: () => process.env.NODE_ENV === 'test'` on the limiter, OR use a very high limit in test env. This avoids any test fragility.
