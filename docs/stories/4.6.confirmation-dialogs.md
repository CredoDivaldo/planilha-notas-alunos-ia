## User Story

Como professora que vai disparar mensagens WhatsApp em massa para todos os estudantes,
quero ver um modal de confirmação antes de o envio ser executado,
para que não envie mensagens por engano e tenha controlo claro sobre a acção irreversível.

---

## Acceptance Criteria

### AC1 — Modal de confirmação antes do envio
Dado que o utilizador preencheu o template e activou a confirmação (ou dry-run),
Quando clica em "Disparar em massa",
Então é apresentado um modal de confirmação com mensagem descritiva antes de qualquer chamada à API.

### AC2 — Implementação com elemento acessível
Dado que o modal é implementado no HTML,
Quando é exibido,
Então usa o elemento `<dialog>` nativo HTML5 ou uma `<div>` com `role="dialog"` e `aria-modal="true"`.

### AC3 — Título descritivo referenciado por aria-labelledby
Dado que o modal tem um título visível,
Quando o modal abre,
Então o elemento `<dialog>` (ou div com role=dialog) tem `aria-labelledby` apontando para o ID do título do modal.

### AC4 — Focus trap dentro do modal
Dado que o modal está aberto,
Quando o utilizador pressiona Tab,
Então o foco cicla apenas entre os elementos interactivos dentro do modal (botão "Confirmar" e botão "Cancelar") e não sai para o resto da página.

### AC5 — Modal fecha com Cancelar, Escape, ou clique fora
Dado que o modal está aberto,
Quando o utilizador clica em "Cancelar", pressiona a tecla Escape, ou clica no overlay fora do modal,
Então o modal fecha sem executar o envio e o foco retorna ao botão "Disparar em massa".

### AC6 — Zero chamadas alert() na aplicação
Dado que toda a validação de formulários e confirmações estão implementadas,
Quando o código fonte é analisado,
Então não existe nenhuma chamada `alert()` em `public/app.js` (confirmado: Story 4.2 já substituiu as chamadas anteriores; esta story garante zero para o futuro também).

### AC7 — Modal acessível: anunciado por screen readers ao abrir
Dado que o modal abre,
Quando um screen reader está activo,
Então o conteúdo do modal é anunciado automaticamente via `aria-live="assertive"` no elemento de diálogo ou pelo mecanismo nativo do `<dialog>` com `.showModal()`.

## Dev Notes

### Estado actual dos alert() em public/app.js

Auditoria em 2026-06-05:
- **Zero chamadas `alert()`** encontradas em `public/app.js`
- Story 4.2 substituiu o `alert("Marca a confirmação...")` por inline error em `#sendConfirmError` (linha 406-413)
- O handler `sendBtn` executa o envio directamente após validação sem modal de confirmação

### Comportamento actual do sendBtn (antes desta story)

```javascript
// Após validação passa, chama directamente a API:
sendResult.textContent = "A enviar...";
const response = await fetch("/api/send/bulk", { ... });
```

A story introduz um passo intermédio: modal de confirmação entre a validação OK e a chamada `fetch("/api/send/bulk")`.

### Padrão de implementação recomendado

```javascript
// showConfirmModal retorna Promise<boolean>
async function showConfirmModal(message) {
  return new Promise((resolve) => {
    // Preenche mensagem no modal
    // Abre com dialog.showModal()
    // Botão Confirmar: resolve(true), fecha modal
    // Botão Cancelar / Escape / backdrop: resolve(false), fecha modal
  });
}

// No sendBtn handler, após validação:
const confirmed = await showConfirmModal("Confirmas o envio de mensagens para todos os estudantes com match?");
if (!confirmed) return;
// ... continua com fetch
```

### Elemento `<dialog>` nativo

`<dialog>.showModal()` fornece:
- Focus trap nativo em browsers modernos
- Tecla Escape fecha automaticamente (evento `cancel`)
- Backdrop nativo (pseudo-elemento `::backdrop`)
- Anúncio por screen readers via `role=dialog` implícito

Para fechar ao clicar no backdrop: comparar `event.target === dialog` no click listener do dialog.

### Decisions Log

1. **Closure ordering para handlers:** `handleBackdropClick` e demais handlers dependentes de `cleanup` foram movidos para dentro do executor `new Promise(...)`, garantindo que `cleanup` está em scope quando os event listeners são adicionados. Bug detectado durante auto-revisão dos testes (ReferenceError: cleanup is not defined).
2. **Focus inicial no botão Cancelar:** Para uma acção destrutiva (envio em massa irreversível), o foco inicial no Cancelar é a escolha mais segura — o utilizador tem de fazer esforço consciente para confirmar.
3. **`<dialog>` nativo com `.showModal()`:** Preferido a `<div role="dialog">` por fornecer focus trap nativo, Escape via evento `cancel`, backdrop com `::backdrop`, e `aria-modal` semântico implícito. jsdom 20 suporta `HTMLDialogElement` mas não `.showModal()` — polyfill mínimo de `open/close` adicionado aos testes.
4. **`onNativeCancel` com `event.preventDefault()`:** Previne o comportamento padrão do browser de fechar o dialog antes de `cleanup()` resolver a Promise (evita race condition entre close nativo e close gerido).
5. **Estilos em `styles.css` com tokens de Story 4.4:** Todos os valores de padding, spacing e cor usam `var(--*)`. `::backdrop` usa `rgba` directo (sem token para transparência) — decisão aceitável pois não há token de opacity definido.
